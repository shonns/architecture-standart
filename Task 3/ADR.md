#### Логика выбора технологий

- Новый сервис вынесен отдельно от ИБ-монолита, чтобы не зависеть от релизного цикла подрядчика и снизить риски регресса.
- Java Spring Boot для DepositService — унификация со стеком существующей системы кол-центра и компетенциями команды (есть Java-специалисты).
- PostgreSQL — быстрая поставка, низкая стоимость, зрелая репликация, соответствие нефункциональным требованиям.
- API Gateway — единая точка контроля безопасности, лимитов и маршрутизации публичного и внутреннего трафика.
- Ручная интеграция с АБС в MVP — минимизация сроков; автоматизацию интеграции с АБС (API/шины, PL/SQL API) запланировать на следующий этап.
- ELK/OpenSearch, Prometheus/Grafana — уже применяются или легко внедряются в on-prem ландшафт; обеспечивают аудит и наблюдаемость.
- СМС через существующий шлюз — использование уже поддерживаемого канала уведомлений.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**

- Реализовать функционал в составе монолита ИБ: быстрее на старте, но сильная зависимость от подрядчика и риски регресса.
- Хранить заявки в MS SQL (совместно с ИБ): меньше новых компонентов, но усиливает связанность и операционные риски для монолита.

**Недостатки, ограничения, риски**

- Ручное участие бэк-офиса в MVP: операционные затраты, возможные ошибки переноса данных из/в АБС.
- Дублирование данных (заявка в PostgreSQL, договор в АБС) до внедрения полноценной интеграции и согласования справочников.
- Сложность и уязвимость прямой интеграции ИБ с новым API (версионирование, сессии, единая SSO отсутствует).
- Публичный endpoint для сайта требует усиленной защиты (боты, DDoS атаки).
- Управление персональными данными: необходимо строгое разграничение доступа, шифрование, журнал согласий; риски несоответствия 152-ФЗ при неверной конфигурации.
- Зависимость от команд подрядчиков (ИБ и платформа кол-центра) при выпускаемых изменениях в UI и расширениях.
